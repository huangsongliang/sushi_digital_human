## 一、产品定位与差异化

| 维度 | 说明 |
|------|------|
| 核心价值 | 将几百页卷宗 → 10 分钟内产出结构化诉讼分析报告 |
| 目标用户 | 中小律师事务所（10-50 人规模）、企业法务部门 |
| 差异化 | 聚焦「诉讼策略分析」而非「法律检索」；证据链可视化而非纯知识图谱 |
| 竞争参考 | 秘塔 AI（法律检索）、北大法宝（数据库+AI）、无讼（社区+案例） |
| 我们的优势 | 端到端流水线（上传→分析→报告），减少律师手动整理时间 |

## 二、三阶段实施计划

### 第一阶段：MVP 验证（2-3 周）

**目标**：能跑通一个简单借贷纠纷案件的全链路

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ PDF 上传  │ →  │ OCR 识别  │ →  │ NER 抽取  │ →  │ 图谱构建  │ →  │ 报告生成  │
│ (PyMuPDF) │    │ (EasyOCR) │    │(few-shot) │    │(人物-证据)│    │(模板填槽) │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

| 功能模块 | 技术方案 | 工作量 |
|----------|----------|--------|
| PDF 解析 | PyMuPDF（已有）+ EasyOCR 扫描件 | 0.5 天（调优） |
| NER 抽取 | 通用 NER + LLM few-shot（通义千问） | 2 天（Prompt 工程） |
| 关系抽取 | 定义 10 种法律关系 schema + Prompt | 2 天 |
| 图谱构建 | 现有 Neo4j 连接器（Mock 模式） | 1 天 |
| 证据链可视化 | ECharts 力导向图（前端） | 2 天 |
| 报告生成 | 法律模板 + LLM 填槽 | 2 天 |
| 前端页面 | Vue 3 法律专用页面 | 3 天 |
| **合计（含联调）** | | **约 3 周** |

**不包含**：
- ❌ 判例检索（用通用网络搜索替代）
- ❌ 多 Agent 协作
- ❌ 协作/批注功能
- ❌ Word/PDF 导出

**验证标准**：找一份真实借贷纠纷案卷（100 页以内），上传后在 10 分钟内产出分析报告，律师评价"有用 / 节省了 2 小时以上"。

### 第二阶段：产品化（4-6 周）

**目标**：能给 3-5 个律师朋友付费试用

| 新增功能 | 详细说明 |
|----------|----------|
| 判例库 | 手动录入 100 个典型案例（借贷/合同/劳动 各 30+），按三级权威分类，向量化入库 |
| 法律 NER 升级 | 训练/微调法律专用 NER 模型（可选），或持续优化 few-shot Prompt |
| 证据链可视化 | 独立页面，展示「证据 → 事实主张 → 法律依据 → 诉求」的完整链路 |
| 报告导出 | Word（可编辑）+ PDF（不可编辑），符合法院格式 |
| 多轮对话 | 在案件上下文中追问："这个证据有什么风险？""换一个诉讼策略呢？" |
| 多 Agent | 原告分析 Agent + 被告分析 Agent + 法官视角 Agent（三方博弈视角） |

**验证标准**：3 个律师愿意付钱试用（哪怕每月 500 元），NPS ≥ 30。

### 第三阶段：商业化（8-12 周）

**目标**：能持续产生收入

| 新增功能 | 详细说明 |
|----------|----------|
| 判例库自动更新 | 接入判例数据源（付费 API 或合规爬虫） |
| 多用户协作 | 案件共享、批注评论、版本管理 |
| 计费/订阅 | 按案件收费（¥500/案）或 SaaS 月费（¥2000/月） |
| 私有化部署 | Docker 一键部署，支持离线运行（数据不出律所内网） |
| 移动端适配 | 律师在法庭/客户现场快速查看案件摘要 |

**验证标准**：10 个付费用户，或月经常性收入 ≥ 2 万。

## 三、关键技术架构决策

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────┐
│                     前端 (Vue 3)                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────────┐ │

│  └──────────┘ └──────────┘ └──────────────────────┘ │
└──────────────────────┬──────────────────────────────┘
                       │ REST API
┌──────────────────────┴──────────────────────────────┐
│                  后端 (FastAPI)                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────────┐ │
│  │ OCR 服务  │ │ NER 服务  │ │ 报告生成服务         │ │
│  │ PyMuPDF  │ │ LLM Few- │ │ 模板引擎 + LLM       │ │
│  │ EasyOCR  │ │  Shot    │ │                      │ │
│  └────┬─────┘ └────┬─────┘ └──────────┬───────────┘ │
│       │            │                  │              │

│  │              RAG 检索层（已有）                    │  │
```

### 3.2 核心决策记录

| 决策项 | 选择 | 原因 |
|--------|------|------|
| OCR 引擎 | EasyOCR（当前）→ PaddleOCR（升级） | EasyOCR 已集成；PaddleOCR 对中文法律文档识别率更高 |
| NER 方案 | LLM few-shot（本阶段） | 不引入额外模型依赖，快速验证；后期可微调专用模型 |
| 图谱存储 | Neo4j（生产） | 关系型查询（如"找到所有指向同一被告的证据"）远超向量数据库 |
| 判例检索 | 本阶段不实现 | 数据源是最大瓶颈；先用通用搜索验证需求真实性 |
| 报告生成 | 固定模板 + LLM 填槽 | 法律报告有固定格式要求，自由生成风险高 |
| Agent 模式 | 单 Agent（本阶段） | MVP 先验证核心链路，多 Agent 是锦上添花 |

### 3.3 数据模型概览

```python
# 法律案件核心实体
class LegalCase:
    case_id: str
    case_type: str          # 案由（借贷纠纷/合同纠纷/劳动仲裁...）
    plaintiff: Party        # 原告
    defendant: Party        # 被告（支持多被告 list）
    third_parties: list[Party]  # 第三人
    court: str              # 受理法院
    filing_date: date
    claims: list[Claim]     # 诉讼请求
    defenses: list[Defense] # 答辩意见
    evidence: list[Evidence] # 证据清单
    applicable_laws: list[LegalArticle]  # 适用法条

class Party:
    role: str               # PLAINTIFF / DEFENDANT / THIRD_PARTY
    name: str
    agent: Optional[str]    # 代理人/律师
    type: str               # 自然人 / 法人 / 非法人组织

class Evidence:
    evidence_id: str
    type: str               # 书证/物证/证人证言/电子数据...
    description: str
    supports: list[str]     # 支撑的事实主张 ID
    credibility: str        # 证据效力评级（高/中/低/待核实）

class LegalArticle:
    law_name: str           # 法律名称（如"民法典"）
    article_number: str     # 条文编号
    content: str            # 条文内容
    source: str             # 来源标注
```

## 四、风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 判例数据源不可得 | 高 | 高 | 第一阶段不依赖判例；验证需求后购买付费 API |
| OCR 准确率低（扫描件质量差） | 中 | 中 | 对低置信度区域标记"需人工核实"；换 PaddleOCR |
| LLM 幻觉（编造法条） | 中 | 高 | Prompt 强制约束 + 法条来源校验 + 人工审核出口 |
| 律师不信任 AI | 高 | 中 | 报告标注"AI 辅助生成，仅供参考"；提供溯源链接 |
| 客户数据安全合规 | 低 | 高 | AES-256 加密、不可删除审计日志、数据不出境 |
| 竞争产品快速跟进 | 中 | 中 | 聚焦差异化（诉讼策略 > 法律检索），快速占领中小律所 |
